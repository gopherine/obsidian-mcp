# obsidian-kb v0.2 ‚Äî LLM-Ready Knowledge Base

**Date**: 2026-03-17
**Status**: Design approved
**Scope**: 4 new subsystems to make the vault useful for autonomous LLM sessions

## Problem

v0.1 built the infrastructure (CLI, MCP server, vault structure, security) but the vault has no useful content. 7 of 8 projects have stub context.md with "Tech stack: To be documented." There's no task tracking, no learning persistence, and no session history. An LLM starting a session gets almost nothing from the vault.

## What's Being Added

### 1. Auto-Discovery (`vault_init`)

Scans a git repo and generates a draft context.md. Does NOT auto-write ‚Äî prints draft for human approval.

**Scans for**: README.md, AGENTS.md, CLAUDE.md, go.mod, Cargo.toml, package.json, pyproject.toml, requirements.txt, Makefile, Dockerfile, docker-compose.yml.

**Detects**: language, framework, project structure (monorepo/microservices/single), dependencies.

**Git activity**: `git log --oneline -20` with 5-second timeout. Bounded output.

**CLI**: `obsidian-kb init <project-path> [--slug <name>]`
**MCP**: `vault_init(project_path, slug?) ‚Üí { draft_context_md, detected_tech_stack }`

**Output format**: Complete context.md with frontmatter, populated sections (What It Is, Current State, Tech Stack, Key Files, Recent Activity).

**Security ‚Äî external filesystem access**:
`init` is the ONLY command that reads outside the vault. It uses a dedicated `safeExternalPath(path)` validator (not VaultFS) that enforces:
- Path must be under `$HOME` (same `homedir() + "/"` check as config.ts)
- Path must not contain `..` or symlinks escaping home
- Path must be an actual git repo (`git rev-parse --git-dir` succeeds)
- Reading uses raw `fs.readFile` (not VaultFS) ‚Äî VaultFS is only for vault writes
- The draft output is returned as a string, never written to vault automatically

### 2. Task Board (replaces todos.md)

Each task is its own markdown file in `projects/<slug>/tasks/`. Queryable, linkable, visible in Obsidian graph.

**Task file naming**: `task-NNN-<slug>.md` where NNN is zero-padded, auto-incremented per project by scanning existing files in `projects/<slug>/tasks/` (same pattern as `decide.ts` ADR numbering). The ID is `task-NNN`.

**Task frontmatter**:
```yaml
---
type: task
project: <slug>
status: backlog | in-progress | blocked | done | cancelled
priority: p0 | p1 | p2
blocked_by: []                     # array of task IDs, e.g. ["task-001"]
assigned_to: claude-code | opencode | codex | human
sprint: ""
created: YYYY-MM-DD
updated: YYYY-MM-DD
tags: []
---
```

**`blocked_by` semantics**: Advisory only. Tasks CAN be moved to `in-progress` even if blockers aren't `done`. The `task board` output marks blocked tasks with a `[BLOCKED]` indicator. `task list --status blocked` returns tasks where any blocker is not `done`.

**CLI commands**:
- `obsidian-kb task list [--project X] [--status Y] [--priority Z]`
- `obsidian-kb task add "title" --priority p0 [--blocked-by task-001] [--assigned-to claude-code]`
- `obsidian-kb task update <id> --status done`
- `obsidian-kb task board [--project X]` ‚Äî kanban view grouped by status

**MCP tool `vault_task` ‚Äî per-action parameters**:

| Action | Required | Optional | Returns |
|--------|----------|----------|---------|
| `add` | `title` | `priority` (default p1), `blocked_by`, `assigned_to`, `project`, `tags` | `{ task_id, path }` |
| `list` | ‚Äî | `project`, `status`, `priority`, `assigned_to` | `{ tasks: [{ id, title, status, priority, blocked_by, assigned_to }] }` |
| `update` | `task_id` | `status`, `priority`, `blocked_by`, `assigned_to`, `title` | `{ task_id, updated_fields }` |
| `board` | ‚Äî | `project` | `{ backlog: [...], in_progress: [...], blocked: [...], done: [...] }` |

**Migration from todos.md**: Existing todo entries are converted to task files. Priority mapping: `high ‚Üí p0`, `medium ‚Üí p1`, `low ‚Üí p2`. Completed todos become tasks with `status: done`.

**`vault_todo` deprecation**: `vault_todo` remains in MCP ListTools for v0.2 with description: "[Deprecated ‚Äî use vault_task] ...". Internally delegates to `vault_task`: `add` ‚Üí task add with p1 default, `list` ‚Üí task list, `complete` ‚Üí task update --status done. Removed in v0.3.

### 3. Learning Capture (`vault_learn`)

Explicit command for LLMs to persist discoveries. Called when an LLM learns something worth remembering across sessions.

**Stored in**: `projects/<slug>/learnings/NNN-<slug>.md` where NNN is auto-incremented per project (same as tasks/decisions).

**Slug derivation**: From a required `title` parameter, slugified via `.toLowerCase().replace(/[^a-z0-9]+/g, "-")`.

**Learning frontmatter**:
```yaml
---
type: learning
project: <slug>
status: active
confidence: high | medium | low
source: claude-code | opencode | codex
session_id: <id>                   # links learning to the session that captured it
created: YYYY-MM-DD
tags: []
---
```

**CLI**:
- `obsidian-kb learn --title "Auth middleware location" --discovery "Lives at internal/middleware/auth.go" [--project X] [--tags a,b] [--confidence high]`
- `obsidian-kb learn list [--project X] [--tag auth]`

**`learn list` output**: `{ learnings: [{ id, title, confidence, tags, created, path }] }` (JSON). CLI renders as table: `[001] [high] Auth middleware location (auth, middleware) ‚Äî 2026-03-17`.

**MCP tool**: `vault_learn(action, title?, discovery?, project?, tags?, confidence?)`

| Action | Required | Optional | Returns |
|--------|----------|----------|---------|
| `add` | `title`, `discovery` | `project`, `tags`, `confidence` (default medium) | `{ learning_id, path }` |
| `list` | ‚Äî | `project`, `tag` | `{ learnings: [{ id, title, confidence, tags, created, path }] }` |

**Bootstrap**: Tier 1 includes learning count. Tier 2 returns full list.

### 4. Session History

Extend `vault_session complete` to persist session summaries as vault notes.

**Stored in**: `projects/<slug>/sessions/YYYY-MM-DD-<tool>-<short-id>.md`

**Session frontmatter**:
```yaml
---
type: session
project: <slug>
tool: claude-code | opencode | codex
session_id: <id>
status: completed
started_at: ISO8601
completed_at: ISO8601
outcome: "What was accomplished"
files_touched: []
tasks_completed: []
learnings_captured: 0
---
```

**`files_touched`**: Relative paths to project root (opaque strings, no validation ‚Äî matches v0.1 behavior).
**`tasks_completed`**: Task IDs (e.g. `["task-001"]`). No validation that they exist ‚Äî advisory metadata.

**CLI change**: `obsidian-kb session complete <id> --outcome "text" [--files f1,f2] [--tasks task-001]`

**MCP change**: Extend `vault_session` action="complete" to accept `outcome`, `files_touched`, `tasks_completed`.

**Bootstrap**: Tier 1 includes "Last session (2h ago): <outcome>".

## Bootstrap Integration Changes

`contextCommand` gains two new fields in its return type:

```typescript
interface ContextResult {
  // existing fields...
  learning_count: number;                                    // NEW
  last_session: { outcome: string; completed_at: string } | null;  // NEW
}
```

These are populated by scanning `projects/<slug>/learnings/` (count files) and `projects/<slug>/sessions/` (find most recent by filename sort, read frontmatter).

`hooks/bootstrap.sh` reads these from the context output and appends to Tier 1:
```
## Learnings: 12 available (use vault_learn list)
## Last Session (2h ago): Implemented domain mapping API endpoint
```
